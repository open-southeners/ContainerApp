# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

ContainerApp is a native macOS SwiftUI menu-bar app + dashboard window that wraps Apple's
`container` CLI (1.0.0+). It lists/inspects/controls containers by shelling out to the CLI —
there is no daemon or library binding. Requires macOS 26+ (Apple's `container` runtime requirement); App Sandbox
is **off** because the app must spawn the CLI and reach its XPC apiserver, so distribution is
Developer ID, not Mac App Store.

## Build & test

The Xcode project is generated by XcodeGen from `project.yml` and is **gitignored** — never edit
`ContainerApp.xcodeproj` by hand; edit `project.yml` and regenerate. New source files are picked
up automatically (the target globs the `ContainerApp/` folder), but you must regenerate after
adding/removing files or changing build settings.

```bash
xcodegen generate                                              # after touching project.yml or file layout
xcodebuild -scheme ContainerApp build                          # build
xcodebuild -scheme ContainerApp -destination 'platform=macOS,arch=arm64' test   # run all tests
```

Pass `-destination 'platform=macOS,arch=arm64'` to any scripted build/test — without it
`xcodebuild` warns and non-deterministically picks among arm64/x86_64/Any Mac.

Tests use **Swift Testing** (`import Testing`, `@Suite`/`@Test`/`#expect`), not XCTest. Run one
suite with `-only-testing:ContainerAppTests/<SuiteOrTestName>`.

- `ContainerCLIModelsTests` / `StatsMergeTests` / `TerminalLauncherTests` — pure unit tests,
  always run. Decoder tests load real CLI output from `Fixtures/` (copied into the test bundle as
  a folder reference).
- `ContainerCLIRuntimeIntegrationTests` — gated behind a synchronous availability check
  (CLI present **and** `container system status` exits 0); skips cleanly otherwise and creates
  real scratch containers when it runs.

## Toolchain constraints

- Swift 6 language mode, `SWIFT_STRICT_CONCURRENCY: complete`. All types crossing concurrency
  boundaries must be `Sendable`; mutable shared state uses actors or lock-guarded `@unchecked
  Sendable` reference types (see `ProcessRunner.PipeBuffer`).
- SwiftUI uses the Observation framework: view model is `@Observable @MainActor`, injected via
  `.environment(model)` and read with `@Environment` — **not** `ObservableObject`/`@StateObject`.

## Architecture

Data flows in one direction: **View → `ContainersViewModel` → `ContainerRuntime` → CLI**, with
decoded results flowing back up.

- **`ContainerRuntime`** (`Runtime/ContainerRuntime.swift`) — `Sendable` protocol, the seam the
  whole app is built around. All methods `async throws`. Two implementations:
  - `ContainerCLIRuntime` — the real one. Shells out via `ProcessRunner`. Handles binary
    discovery (UserDefaults `containerCLIPath` override → init override → `/usr/local/bin` →
    `/opt/homebrew/bin` → `which`), cached in a `BinaryCache` actor keyed by the override string
    so editing Settings invalidates the cache.
  - `MockContainerRuntime` — canned data; swap it into `ContainerAppApp` for UI work without a
    live CLI.
- **`ProcessRunner`** (`Runtime/ProcessRunner.swift`) — wraps `Foundation.Process`. Reads stdout
  and stderr concurrently on background threads (so a chatty process can't deadlock on a full
  pipe buffer) and sets stdin to `/dev/null` so interactive CLI prompts fail fast instead of
  hanging. **Never throws on non-zero exit** — returns exit code + stderr in `ProcessResult` for
  the caller to interpret.
- **Error mapping** — `ContainerCLIRuntime.runChecked` turns non-zero exits into
  `ContainerRuntimeError`: stderr containing `XPC connection error` / `container system start`
  → `.systemNotRunning`; "not found" → `.notFound`; binary discovery failure → `.cliNotFound`.
  `systemStatus()` deliberately does *not* use `runChecked` so a stopped system reads as
  `.stopped` rather than an error. The view model's `handle(_:)` branches on these to drive the
  UI (install message vs. Start-System prompt vs. error banner) — raw stderr is never shown.
- **DTO layer** (`Runtime/ContainerCLIModels.swift`) — `CLIContainerDTO`/`CLIStatsDTO` mirror the
  real nested JSON with all-optional fields (tolerates unknown future keys), then map to the flat
  UI models (`ContainerSummary`, `ContainerStats`). `FlexibleContainerDecoder` is the only decode
  entry point; DecodingErrors become `.decodingFailed` with a bounded preview. The container
  **id doubles as its name** (the CLI has no separate name field).
- **Stats merge** (`ContainersViewModel.mergeStats`) — the CLI reports cumulative `cpuUsageUsec`,
  **no CPU percentage**. CPU% is derived in the view model from the delta between two samples
  (`cpuPercent` static helper, kept pure/`nonisolated` for testing). CPU/mem text on
  `ContainerSummary` is filled by merging `stats` results by id — never from `list`. The dashboard
  polls every 5s via a `quiet` refresh (no loading spinner) in a `.task` that cancels on close.
- **`TerminalLauncher`** (`Utilities/`) — opens a shell via osascript→Terminal.app. Two layers:
  `shellCommand(forContainerID:)` is pure/testable and **validates the id against
  `^[A-Za-z0-9._-]+$` (and rejects leading `-`) before interpolating it unquoted** into the shell
  line — this is injection defense, keep it. `open(command:)` owns the AppleScript side effect;
  first use triggers the macOS Automation consent prompt (declared via
  `NSAppleEventsUsageDescription` in Info.plist).
- **Compose layer** — compose is a distinct binary with its own seam (`Runtime/ComposeRuntime.swift`):
  - **`ComposeRuntime`** protocol — `up`/`build`/`version` only. `down` is intentionally absent
    (see Compose facts below); stop/teardown is handled via `ContainerRuntime.stop(id:)` on
    matched containers in reverse YAML service order.
  - **`ContainerComposeCLIRuntime`** — shells out to `container-compose`. Binary discovery mirrors
    `ContainerCLIRuntime` with its own `BinaryCache` keyed by UserDefaults
    `containerComposeCLIPath` override → init override → `/usr/local/bin/container-compose` →
    `/opt/homebrew/bin/container-compose` → `which`. Failure throws
    `ContainerRuntimeError.composeCLINotFound` (distinct from `.cliNotFound` so the UI shows the
    brew install hint). Every command sets `currentDirectoryURL` to the compose file's parent
    folder and passes `-f <filename>`. ANSI escape codes and `\r` progress rewrites are stripped
    from stdout before the output is surfaced.
  - **`ComposeFileParser`** (`Utilities/ComposeFileParser.swift`) — uses **Yams** (the app's first
    SwiftPM dependency, declared in `project.yml`). Pure function `parse(text:fileURL:)` decodes
    only `name:` + service keys + `image`; all fields lenient/optional (same policy as
    `CLIContainerDTO`). Service values can be null (bare key). Parse failures throw
    `ComposeError.parseFailed` with a 200-char preview. Thin `load(fileURL:)` wrapper reads from
    disk and sets `isMissing = true` when the file is unreadable.
  - **`ComposeProjectStore`** (`Utilities/ComposeProjectStore.swift`) — persists registered
    compose-file paths as JSON-encoded `[String]` under UserDefaults key `composeProjectPaths`
    (takes a `UserDefaults` instance at init; tests use a suite-named instance). Knows paths only;
    reparsing into `ComposeProject` values is the view model's responsibility.

## CLI facts worth knowing

- `container list --all --format json` — `--all` is required to include stopped containers.
  Each element's `status` is an **object** (`{state, startedDate}`), not a string.
- `container stats --format json --no-stream` — only running containers appear; cumulative usec,
  no percentage.
- `container inspect <id>` — JSON by default, no `--format` flag; returns an array.
- `container image list --format json` — each element has `configuration.name` (fully-qualified
  ref, same form as containers' `configuration.image.reference` — exact string match works for
  in-use detection) and per-arch `variants`. **`configuration.descriptor.size` is the manifest
  size (~9 KB), not the image size** — real weight is the sum of `variants[].size`. Multi-arch
  images include `architecture: "unknown"` attestation shims — filter them for display.
- `container image inspect <ref>` — takes name refs, **no `--format` flag**, prints pretty JSON
  (an array of the same element shape as `image list`).
- `container image prune` — dangling images only (`--all` would cover all unused); prints a
  summary line on stdout ("Reclaimed … in disk space") which the UI surfaces verbatim.
- When the apiserver is stopped, every command exits non-zero with `XPC connection error` and the
  hint `container system start`.
- First-ever `container system start` needs a kernel and prompts interactively (fails under our
  `/dev/null` stdin); the fix is a one-time `container system kernel set --recommended`.

Captured real CLI output lives in `Fixtures/` (see `Fixtures/README.md` for exact shapes) and
backs the decoder tests — re-capture if the CLI version changes.

## Compose facts worth knowing

- Subcommands: `up`, `down`, `build`, `version` (plus `--version`). No `ps`, `logs`, `restart`,
  `pull`, or `stop` — service status comes from `container list` grouping.
- `up` flags used: `-d` (always detached), `-f <filename>`, `-b` (rebuild), `--no-cache`.
- **Container naming**: `<project>-<service>` where project = compose `name:` field if present,
  else the **cwd folder's last path component** with `.` replaced by `_`. Because the project name
  derives from the process cwd (not the `-f` path), the app always sets `currentDirectoryURL` to
  the compose file's parent folder *and* passes `-f <filename>` — never pass an absolute path
  without cwd.
- **`down` is broken** in the brew formula (0.12.0 vs runtime 1.0.0): XPC protocol mismatch
  (`DecodingError.typeMismatch … Path: signal`), containers keep running. The app therefore stops
  containers itself via `ContainerRuntime.stop(id:)` and never calls `container-compose down`.
  Revisit when the brew formula reaches 1.0.0+.
- Install: `brew install container-compose` (formula name `container-compose`). As of 2026-06-12
  the formula ships **0.12.0** while GitHub is at **1.0.0** — do not hard-require a version;
  `--version` prints `container-compose version X.Y.Z`, captured for display only.
- Stopped apiserver: `up` exits 1 with stderr `Error: XPC connection error: Connection invalid`.
  The existing `XPC connection error` phrase mapping covers this; the `container system start`
  hint does **not** appear (unlike the `container` CLI).
- stdout from `up`/`build` contains ANSI escape codes (Rainbow) and `\r`-rewritten progress lines.
  `ContainerComposeCLIRuntime` strips both before storing the output. Pipes usually disable color
  but the stripping is unconditional.
- No shell spawned (argv array), so service names from YAML need no quoting or validation.
